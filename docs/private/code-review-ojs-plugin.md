## Critical — must fix before deploy

### C1. Preflight response field name mismatch with WP plugin

**File:** `WpojsApiController.php` lines 177, 178, etc. (all `$checks[]` entries)
**Severity:** Integration bug — WP plugin will display blank check names

The OJS plugin returns preflight check items with the key `check`:

```php
$checks[] = ['check' => "Repo::user()->{$method}()", 'ok' => $ok];
```

But the WP plugin reads the key `name`:

```php
// class-wpojs-api-client.php:145
$failed_checks[] = $check['name'];

// class-wpojs-cli.php:426
WP_CLI::log( sprintf( '    %s %s', $icon, $check['name'] ) );
```

**Impact:** The WP "Test connection" button and `wp ojs-sync test-connection` CLI command will show blank/empty check names. Failed compatibility checks will report empty strings instead of meaningful diagnostics. The `compatible` boolean still works, so it won't cause sync failures, but operators cannot diagnose *which* check failed.

**Fix:** Either rename `check` to `name` in the OJS plugin, or rename `name` to `check` in the WP plugin. Recommend changing the OJS plugin since the WP plugin's `name` is the more conventional field name:

```php
$checks[] = ['name' => "Repo::user()->{$method}()", 'ok' => $ok];
```

This affects every `$checks[]` assignment in `preflight()` (approximately 20 occurrences).

### C2. XSS vulnerability in login hint via `innerHTML`

**File:** `WpojsSubscriptionApiPlugin.php` line 92
**Severity:** Stored XSS risk (low likelihood but needs fixing)

The login message uses `innerHTML` to inject content:

```php
div.innerHTML = "' . __('plugins.generic.wpojsSubscriptionApi.loginHint', ['lostPasswordUrl' => $lostPasswordUrl]) . '";
```

The `$lostPasswordUrl` is generated by OJS's dispatcher and is generally safe. However, this pattern is fragile:

1. The URL is interpolated into a JavaScript string literal without escaping for JS context. If the URL ever contains a single quote, double quote, or backslash, the JS breaks.
2. The result is passed to `innerHTML`, so if the translated string or URL substitution ever contains unexpected HTML, it will be rendered.

The paywall hint and footer message use `$output .=` concatenation with `htmlspecialchars()` on config values, which is the correct approach. The login hint is the outlier.

**Fix:** At minimum, escape the URL for JS string context:

```php
$escapedUrl = str_replace(["\\", "'", '"', "\n", "\r"], ["\\\\", "\\'", '\\"', '\\n', '\\r'], $lostPasswordUrl);
```

Or better, use `textContent` + `insertAdjacentHTML` with a pre-escaped HTML string, or set `data-` attributes on a script tag and read them in JS.

## Important — should fix before deploy

### I1. `getJournalId()` will throw an uncaught exception if context is null

**File:** `WpojsApiController.php` line 141
**Severity:** Uncaught null dereference crashes the endpoint with an opaque 500 error

```php
private function getJournalId(): int
{
    return Application::get()->getRequest()->getContext()->getId();
}
```

If `getContext()` returns null (e.g. request to the site-wide context without a journal path), calling `->getId()` will throw a fatal error. The `has.context` middleware *should* prevent this, but:

1. The `ping` endpoint explicitly removes `has.context` middleware, and `ping` doesn't call `getJournalId()` — fine.
2. If a future endpoint is added without `has.context`, or if the middleware is bypassed somehow, this becomes a crash.

**Fix:** Add a null check:

```php
private function getJournalId(): int
{
    $context = Application::get()->getRequest()->getContext();
    if (!$context) {
        throw new \RuntimeException('No journal context');
    }
    return $context->getId();
}
```

Or, since middleware should catch this, at minimum add a docblock noting the dependency on `has.context`.

### I2. `user_settings` table may not have a unique constraint for `insertOrIgnore` dedup

**File:** `WpojsApiController.php` lines 798-803
**Severity:** Dedup may silently fail, allowing duplicate welcome emails

The welcome email dedup relies on `insertOrIgnore`:

```php
$inserted = DB::table('user_settings')->insertOrIgnore([
    'user_id' => $userId,
    'locale' => '',
    'setting_name' => 'wpojs_welcome_email_sent',
    'setting_value' => '1',
]);
```

`insertOrIgnore` translates to `INSERT IGNORE INTO ...` in MySQL. This only deduplicates if there is a unique constraint on `(user_id, locale, setting_name)`. The OJS `user_settings` table structure needs to be verified on the actual deployment:

- In OJS 3.5, `user_settings` typically has a unique index on `(user_id, locale, setting_name)`.
- If this index doesn't exist (or was dropped during an upgrade), `insertOrIgnore` will always insert and return true, defeating the dedup.

**Fix:** Verify the unique constraint exists on the target OJS database. Add this to the `/preflight` checks:

```php
// Check user_settings has the unique constraint needed for welcome email dedup
try {
    $indexes = DB::select("SHOW INDEX FROM user_settings WHERE Key_name = 'user_settings_unique'");
    $ok = !empty($indexes);
} catch (\Exception $e) {
    $ok = false;
}
$checks[] = ['name' => 'user_settings unique constraint', 'ok' => $ok];
```

Alternatively, use a `SELECT ... WHERE` check before the insert as a belt-and-suspenders approach (the current code already does this in `sendWelcomeEmail()` but not in `doSendWelcomeEmail()`).

### I3. `generatePasswordResetHash` is checked in preflight but not used in the actual welcome email code

**File:** `WpojsApiController.php` lines 230-236 (preflight) vs lines 810-827 (doSendWelcomeEmail)
**Severity:** Misleading preflight check; possible wrong token mechanism

The preflight checks for `Validation::generatePasswordResetHash()`:

```php
foreach (['suggestUsername', 'encryptCredentials', 'generatePasswordResetHash'] as $method) {
    $ok = method_exists(Validation::class, $method);
```

But the actual welcome email code uses `AccessKeyManager::createKey()` instead:

```php
$accessKeyManager = new AccessKeyManager();
$accessKey = $accessKeyManager->createKey('LoginHandler', $user->getId(), null, $expiryDays);
```

These are two different token mechanisms in OJS:
- `generatePasswordResetHash()` — HMAC-based, tied to user's `dateLastLogin` (invalidates on login)
- `AccessKeyManager::createKey()` — stored in `access_keys` DB table, expires after N days

The code uses `AccessKeyManager` (correct for this use case), but preflight verifies `generatePasswordResetHash` (unused). The preflight should instead verify `AccessKeyManager::createKey()`.

**Fix:** Replace the `generatePasswordResetHash` check in preflight with a check for `AccessKeyManager::createKey`:

```php
if (class_exists(AccessKeyManager::class)) {
    $ok = method_exists(AccessKeyManager::class, 'createKey');
    $checks[] = ['name' => 'AccessKeyManager::createKey()', 'ok' => $ok];
    if (!$ok) $compatible = false;
}
```

And remove `generatePasswordResetHash` from the Validation methods check (it's not used anywhere).

### I4. Race condition in `findOrCreateUser` — catch block is too broad

**File:** `WpojsApiController.php` lines 377-390
**Severity:** Swallows unexpected errors during user creation

```php
} catch (\Exception $e) {
    // Race condition: another request may have created this user concurrently
    $raceUser = Repo::user()->getByEmail($email, true);
    if ($raceUser) {
        return response()->json([
            'userId' => $raceUser->getId(),
            'created' => false,
        ], Response::HTTP_OK);
    }
    return response()->json(
        ['error' => 'Failed to create user'],
        Response::HTTP_INTERNAL_SERVER_ERROR
    );
}
```

The catch block catches *all* exceptions, not just duplicate-key violations. If `Repo::user()->add()` fails for a completely different reason (e.g. a DB connection drop, a validation error in OJS, a missing required field), the code will:

1. Silently check if the user exists by email
2. If they happen to exist (coincidentally), return success with `created: false`
3. If they don't, return a generic 500 with no diagnostics

This could mask real bugs. A race condition specifically throws a duplicate-key `QueryException`.

**Fix:** Narrow the catch to handle the race condition specifically:

```php
} catch (\Illuminate\Database\QueryException $e) {
    // Duplicate key = race condition: another request created this user
    $raceUser = Repo::user()->getByEmail($email, true);
    if ($raceUser) {
        return response()->json(['userId' => $raceUser->getId(), 'created' => false], Response::HTTP_OK);
    }
    return response()->json(['error' => 'Failed to create user'], Response::HTTP_INTERNAL_SERVER_ERROR);
} catch (\Exception $e) {
    return response()->json(['error' => 'Failed to create user'], Response::HTTP_INTERNAL_SERVER_ERROR);
}
```

### I5. `AccessKeyManager::createKey()` — fourth parameter semantics need verification

**File:** `WpojsApiController.php` lines 822-826
**Severity:** Token may expire in 7 hours instead of 7 days (or vice versa)

```php
$expiryDays = Config::getVar('security', 'password_reset_timeout');
if (!$expiryDays) {
    $expiryDays = 7;
}

$accessKey = $accessKeyManager->createKey(
    'LoginHandler',
    $user->getId(),
    null,
    $expiryDays
);
```

The `AccessKeyManager::createKey()` fourth parameter is documented inconsistently across OJS versions. In some versions it's days, in others it might be interpreted differently. The `docs/ojs-api.md` reference mentions `reset_seconds` (seconds) for `generatePasswordResetHash`, which is a different mechanism entirely.

The config value `password_reset_timeout = 7` is read and passed directly. If `createKey()` expects days, this is correct (7 days). If it expects hours or some other unit, the token will expire at the wrong time.

**Fix:** Verify the `AccessKeyManager::createKey()` parameter semantics on OJS 3.5 staging. Add a comment documenting the expected unit. Consider adding this to the preflight checks with a note about the expected config value.

### I6. GDPR deletion does not clear `datePasswordResetRequested` or access keys

**File:** `WpojsApiController.php` lines 466-501
**Severity:** PII residue after GDPR erasure

The `deleteUser` endpoint anonymises email, username, name, affiliation, biography, orcid, url, phone, and mailing address. But it does not clear:

1. `datePasswordResetRequested` — timestamp of last password reset, set by `doSendWelcomeEmail`. Minor PII exposure.
2. `access_keys` table — if a welcome email was sent, the `access_keys` record links the user ID to the `LoginHandler` context. Should be deleted.
3. Other `user_settings` beyond `wpojs_welcome_email_sent` — there may be OJS-native settings containing PII (e.g. `preferredPublicName`).

**Fix:** Add cleanup for access keys and clear all user_settings:

```php
// Remove access keys
DB::table('access_keys')
    ->where('user_id', $userId)
    ->where('context', 'LoginHandler')
    ->delete();

// Clear password reset timestamp
Repo::user()->edit($user, [
    // ... existing anonymisation ...
    'datePasswordResetRequested' => null,
]);
```

Consider whether a broader `user_settings` wipe is appropriate (delete all settings for this user, not just `wpojs_welcome_email_sent`).

### M7. No Content-Type enforcement on POST/PUT endpoints

**File:** `WpojsApiController.php` — all POST/PUT endpoints

The endpoints read `$request->input()` which works for both form-encoded and JSON bodies. There's no check that the Content-Type is `application/json`. The WP plugin always sends JSON, but an attacker could send form-encoded data. Laravel handles both, so this isn't a security issue, but it's less strict than ideal.

**Fix:** Add a Content-Type check in `checkAuth()` for non-GET requests:

```php
if ($request->method() !== 'GET' && !$request->isJson()) {
    return response()->json(['error' => 'Content-Type must be application/json'], Response::HTTP_BAD_REQUEST);
}
```

### N5. `insertOrIgnore` return value semantics

**File:** `WpojsApiController.php` line 798

```php
$inserted = DB::table('user_settings')->insertOrIgnore([...]);

if (!$inserted) {
    return false;

```

Laravel's `insertOrIgnore` returns the number of affected rows. If the row was inserted, it returns `1` (truthy). If it was a duplicate, it returns `0` (falsy). ### N7. Consider rate limiting the welcome email endpoint

The `/welcome-email` endpoint has dedup protection, but no rate limiting. During bulk sync of ~700 users, this endpoint gets called 700 times in sequence. If the dedup flag gets deleted (due to a mail failure) and bulk sync is re-run, all 700 emails could be re-sent. This is by design (retry-safe), but verify the mail relay (Mailgun/SES/Postmark) can handle the burst without rate limiting or throttling.

---

### N8. No logging within the OJS plugin

The OJS plugin has no internal logging. All logging is done by the WP plugin based on HTTP response codes. If an endpoint fails in a way that returns a clean error response, the WP plugin logs it. But if something fails silently (e.g. subscription not actually updated despite 200 response), there's no OJS-side audit trail.

Consider adding `error_log()` calls for exceptional cases, or using OJS's built-in logging mechanism if available.

### N9. `Validation::encryptCredentials` — first parameter significance

**File:** `WpojsApiController.php` line 358

```php
$user->setPassword(Validation::encryptCredentials($username, bin2hex(random_bytes(16))));
```

The first parameter to `encryptCredentials()` is typically the username, used as a salt in older OJS versions. In newer versions it may be ignored (bcrypt generates its own salt). The random password is 32 hex chars (128 bits of entropy) — this is fine. But verify that the generated hash is valid and that the user can successfully reset their password via the welcome email flow.

### Overall assessment

The plugin is well-structured and follows OJS 3.5 patterns correctly. The endpoint design is clean, idempotent where specified, and the auth model (IP allowlist + role check) provides defence-in-depth. Input validation is thorough across all endpoints. The subscription upsert logic correctly handles the four key scenarios (new, extend, reactivate, no-op).

The two critical issues are both integration bugs rather than fundamental design flaws. C1 (field name mismatch) is a one-line rename across 20 occurrences — easy to fix. C2 (innerHTML XSS) is low-likelihood in practice since the URL comes from OJS's own dispatcher, but should be fixed for defence-in-depth.

The important issues are primarily about edge-case robustness (null context, broad exception handling, GDPR completeness) and verification items (dedup constraint, token expiry unit, preflight accuracy). None are architectural concerns.

**Recommendation:** Fix C1, C2, I3, I4, and I6 before deploy. Verify I2, I5, N1-N4 on staging. The remaining items can be addressed post-launch.
